一、list 的深复制、浅复制、值、引用的问题¶

现在依稀记得当年上 C 语言课程的时候，老师在课堂上反复地讲的、习题上反复作的、考试一定不放过的一个内容是数组的下标和越界问题，另一个就是值和引用的问题。用在这里差不多就是所谓的深复制、浅复制的问题。

深复制、浅复制是现在手头的一本Python参考手册上面的用语，各种教材、网上各种资料用语还不一样。英文资料上面的说法也不太一样。

1. = 操作传递的是引用，append、remove是在当前序列上进行的操作¶

对序列的 = 操作传递的是引用，对其中一个序列的修改同时影响另一个序列。 这一点比较简单，放在这里只为完整性。

2. 浅复制：对值元素传递值、对引用元素传递引用。如copy¶

例如：

二、在遍历序列的同时修改序列¶

前面说到，后来总觉得算法似乎有点问题，是因为回忆起来有一篇文章讲过，在遍历的同时个性list要非常谨慎，因为会出问题。但是落园很巧妙地这样使用没有问题。

在位置4那儿，current_string的重新赋值，不讨论内在开销之类的问题，显然用 current_string.remove(character)会直观一些，但是这样用，就会有真正的问题，也就是在遍历的同时修改了序列，遍历会跳过某些元素。下面有例子。

下面一段例子是后来测试出来的，原因大概也是值和引用的问题。在原序列上的修改会影响遍历，而将变量指向新的序列或者新的地址，不会影响遍历。例子如下：

1. 第一个例子：¶

In [4]:
n=list('abcd')
x=1
for c in n:
    print('第{0}遍循环时 c={1}   n={2}'.format(x,c,n))
    n=n[1:]       #n被指向了新的地址，但原地址数据不变，因此c仍然会照预期地遍历下去
    x+=1          

第1遍循环时 c=a   n=['a', 'b', 'c', 'd']
第2遍循环时 c=b   n=['b', 'c', 'd']
第3遍循环时 c=c   n=['c', 'd']
第4遍循环时 c=d   n=['d']

在上面的例子当中，切片对象传递的是值，估计也是浅复制的范畴。落园也是用类似这种方式遍历（位置1、位置4），能够按照预期地遍历下去，同时还不影响while循环里的下一遍遍历。但是该地址上的数据引用数据会不会变成0被垃圾回收，并被覆盖新的数据，产生不可预测的结果呢？这个测试起来比较麻烦，后面再说。

三、切片的上下标问题（补充：其实是开始和结束的索引，不是正常的上下标概念）¶

这个问题以前想当年看书的时候测试过，但是没有记下来，后来又忘记了（现在智商和记忆力怎么退化这么多呢？）这里再记下来吧。

问题源自源代码的位置2，作者用了 sub_string = current_string[index:len(current_string)]，这样丝毫没有问题，简化成 sub_string = current_string[index:] 也没有问题，但也没什么意义，除了少敲几个字符。

但是问题是：我不知哪里来的直觉总是觉得 a[1:2]应该是包含 a[1]、a[2] 两个元素的，但其实是只包含 a[1] 一个元素的。

所以对于一个len(a)=3的序列，a[3]会越界，但a[0:3]没有问题，并且包含了所有的元素，甚至可以a[0:8]；a的最后一个元素是a[2]，但是a[0:2]会漏掉最后一个元素。

简单来说就是切片是包含上标、不包含下标的，这其实是太简单的常识。

四、set 的操作¶

  • 集合是唯一项的无序集
  • 集合不提供索引或切片操作
  • 放入集合的项目是不可变的

没有太多可说的，直接看例子吧。唯一注意的一就是“减”操作。也就是 位置3

五、测试一下上面提到的数据覆盖问题¶

In [13]:
import gc
x = [i for i in range(10000)]
result = []
for n in x:
    x = [i for i in range(10000,20000)]  #这里应该是在新分配的地址上，原地址的数据不变，理论上不影响 n 的遍历
    result.append(n)                      #所以 result 应该与最初的 x 相等。
    gc.collect()
print( result == [i for i in range(10000)])

True

貌似没有覆盖的问题，所以 n 仍然在原来的数据地址上老老实实地、安全的遍历着。上面的程序运行的很慢，本来就是希望有很大的内存读写，所以内存开销上面的优化也没有意义，或许可以进行其它的优化。
